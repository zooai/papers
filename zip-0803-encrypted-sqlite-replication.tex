\begin{abstract}
We specify a sidecar-based replication protocol for Zoo Network services
that use Hanzo Base (per-org encrypted SQLite).  The system streams
Write-Ahead Log (WAL) frames through \texttt{hanzoai/replicate} to
S3-compatible object storage, encrypts at-rest with \texttt{hanzoai/age}
(X25519 + ChaCha20-Poly1305), and achieves $\leq 1$\,s Recovery Point
Objective (RPO) with seconds-scale Recovery Time Objective (RTO).
Each pod replicates to a distinct S3 prefix keyed by pod identity,
enabling StatefulSet-aware restore.  The design reuses the same
\texttt{ghcr.io/hanzoai/replicate:latest} image deployed across
Liquidity and Lux ecosystems, requiring zero Zoo-specific code.
\end{abstract}

\section{Motivation}

Zoo Network services are migrating from PostgreSQL to Hanzo Base
(per-org encrypted SQLite) following the Web5 local-first architecture
described in the Zoo research program~\cite{web5}.  SQLite provides
sub-microsecond read latency, per-tenant data isolation via
deterministic key derivation, and offline-first operation.

However, SQLite's single-writer model means each service instance
owns its database file.  Without replication:

\begin{enumerate}
\item A pod restart loses all data accumulated since the last
  PersistentVolumeClaim (PVC) snapshot.
\item Disaster recovery requires manual PVC restoration from
  cloud provider snapshots (minutes to hours).
\item Cross-region failover is impossible without an external
  replication mechanism.
\end{enumerate}

This proposal introduces continuous WAL streaming to S3 as the
replication primitive, providing:

\begin{itemize}
\item $\leq 1$\,s RPO via 1-second sync intervals
\item Seconds-scale RTO via init-container restore
\item At-rest encryption via \texttt{hanzoai/age}
\item Zero application code changes (sidecar pattern)
\end{itemize}

\section{Architecture}

\subsection{Components}

\begin{description}
\item[\texttt{hanzoai/replicate}] Litestream-based WAL streaming
  daemon.  Watches SQLite WAL, ships frames to S3.  Runs as a
  sidecar container sharing the data volume with the primary
  service container.

\item[\texttt{hanzoai/age}] X25519 + ChaCha20-Poly1305 file
  encryption.  Used for encrypting WAL snapshots and segments
  at rest in S3.  Compatible with \texttt{luxfi/age} (same
  wire format).

\item[\texttt{hanzoai/s3}] S3-compatible object storage.  Deployed
  as a ClusterIP service within the Zoo K8s cluster
  (\texttt{s3.zoo.svc.cluster.local:9000}).  Backed by
  DigitalOcean Spaces or MinIO depending on environment.

\item[ConfigMap] Shared \texttt{replicate-config} ConfigMap
  containing the Litestream YAML configuration.  All services
  reference the same ConfigMap; pod identity (\texttt{POD\_NAME})
  provides per-replica S3 path namespacing.
\end{description}

\subsection{Data Flow}

\begin{verbatim}
Service writes to SQLite
  -> WAL frames written to shared PVC
  -> Replicate sidecar reads WAL
  -> Frames uploaded to S3 (1s interval)
  -> Optional: age encryption before upload

Pod restart:
  -> Init container runs replicate restore
  -> Fetches latest snapshot + WAL segments
  -> Reconstructs SQLite database
  -> Primary container starts with warm data
\end{verbatim}

\subsection{K8s Sidecar Pattern}

Each service pod contains three components:

\begin{enumerate}
\item \textbf{Init container} (\texttt{replicate-restore}): Runs
  \texttt{replicate restore -if-db-not-exists -if-replica-exists}
  to restore from S3 on cold start.  No-op if database already
  exists on the PVC.

\item \textbf{Primary container}: The service binary (e.g.,
  \texttt{zoo-bridge}, future Base-powered services).  Reads and
  writes \texttt{/app/data/data.db}.

\item \textbf{Sidecar container} (\texttt{replicate}): Runs
  \texttt{replicate replicate} to continuously stream WAL changes
  to S3.  Shares the \texttt{/app/data} volume with the primary.
\end{enumerate}

\subsection{S3 Path Layout}

\begin{verbatim}
s3://zoo-data/
  {pod-name}/
    generations/
      {generation-id}/
        snapshots/
          {index}.snapshot.lz4
        wal/
          {index}.wal.lz4
\end{verbatim}

Each pod (e.g., \texttt{zoo-bridge-0}, \texttt{zoo-bridge-1})
replicates to a distinct prefix.  The \texttt{POD\_NAME}
environment variable is injected via the Kubernetes downward API
(\texttt{metadata.name} fieldRef).

\section{Specification}

\subsection{Replicate Configuration}

\begin{lstlisting}[language=yaml,caption={replicate-config ConfigMap}]
dbs:
  - path: /app/data/data.db
    replicas:
      - type: s3
        bucket: zoo-data
        path: ${POD_NAME}
        endpoint: ${LITESTREAM_S3_ENDPOINT}
        region: sfo3
        force-path-style: true
        sync-interval: 1s
        snapshot-interval: 1h
\end{lstlisting}

\begin{itemize}
\item \texttt{sync-interval: 1s} --- WAL frames shipped every
  second, achieving $\leq 1$\,s RPO.
\item \texttt{snapshot-interval: 1h} --- Full snapshot every hour
  bounds restore time (restore replays at most 1 hour of WAL).
\item \texttt{force-path-style: true} --- Required for
  DigitalOcean Spaces and MinIO.
\end{itemize}

\subsection{Encryption}

WAL segments and snapshots are encrypted at rest using
\texttt{hanzoai/age} with per-environment X25519 key pairs:

\begin{lstlisting}[language=yaml,caption={Encryption configuration}]
dbs:
  - path: /app/data/data.db
    replicas:
      - type: s3
        bucket: zoo-data
        path: ${POD_NAME}
        encryption:
          type: age
          key: ${AGE_SECRET_KEY}
\end{lstlisting}

Key management:
\begin{itemize}
\item Encryption keys stored in Zoo KMS (K-Chain) or
  Kubernetes secrets (bootstrapping).
\item One key pair per environment (devnet, testnet, mainnet).
\item Key rotation: generate new pair, re-snapshot with new key,
  update secret.  Old WAL segments remain readable with old key
  until next snapshot.
\end{itemize}

\subsection{Recovery Guarantees}

\begin{description}
\item[RPO $\leq 1$\,s] WAL frames synced every second.
  In the worst case, the last second of writes is lost.

\item[RTO $\approx$ seconds] Init container restores latest
  snapshot ($\leq 1$ hour old) plus WAL replay.  Typical
  restore: 2--5 seconds for databases under 1\,GB.

\item[Durability] S3 provides 11 nines of durability.
  Snapshots are LZ4-compressed and optionally age-encrypted.

\item[Consistency] SQLite WAL mode guarantees atomic commits.
  Replicate ships complete WAL frames, preserving transaction
  boundaries.
\end{description}

\section{Implementation}

\subsection{K8s Patches}

Patches are applied as standalone YAML files in
\texttt{k8s/patches/} rather than inline kustomize overlays.
This allows selective application per-service and per-environment.

Required patches:
\begin{enumerate}
\item \texttt{replicate-config.yaml} --- Shared ConfigMap
\item \texttt{replicate-bridge.yaml} --- Bridge deployment patch
  (when migrated to SQLite)
\end{enumerate}

\subsection{S3 Service}

Deploy \texttt{hanzoai/s3} as a StatefulSet with block storage:

\begin{lstlisting}[language=yaml,caption={S3 service (MinIO)}]
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: s3
spec:
  replicas: 1
  template:
    spec:
      containers:
        - name: s3
          image: ghcr.io/hanzoai/s3:latest
          ports:
            - containerPort: 9000
          env:
            - name: MINIO_ROOT_USER
              valueFrom:
                secretKeyRef:
                  name: s3-credentials
                  key: ACCESS_KEY
            - name: MINIO_ROOT_PASSWORD
              valueFrom:
                secretKeyRef:
                  name: s3-credentials
                  key: SECRET_KEY
\end{lstlisting}

\subsection{Apply Script}

A shell script (\texttt{scripts/apply-replicate.sh}) orchestrates
the full deployment:

\begin{enumerate}
\item Verifies \texttt{s3-credentials} secret exists
\item Applies the shared ConfigMap
\item Patches each target service
\item Restarts patched workloads
\item Waits for rollout completion
\end{enumerate}

\midrule

WAL-to-S3 replication provides the best tradeoff for Zoo's
requirements: near-zero RPO, fast recovery, minimal operational
complexity, and compatibility with the Base/SQLite-per-org
architecture.

\section{Cross-Ecosystem Compatibility}

The \texttt{ghcr.io/hanzoai/replicate:latest} image is shared
across all ecosystems:

\begin{itemize}
\item \textbf{Liquidity}: ATS, BD, TA, IAM, KMS (5 services)
\item \textbf{Lux}: Future Base-powered services
\item \textbf{Zoo}: Bridge and future Base-powered services
\end{itemize}

The only per-ecosystem differences are:
\begin{itemize}
\item S3 bucket name (\texttt{liquidity-data} vs \texttt{zoo-data})
\item S3 endpoint (GCS vs DigitalOcean Spaces)
\item Region (\texttt{us-central1} vs \texttt{sfo3})
\item Age encryption key pairs
\end{itemize}

All configuration is injected via ConfigMap and secrets.
Zero code divergence between ecosystems.

\section{Security Considerations}

\begin{enumerate}
\item \textbf{S3 credentials}: Stored in Kubernetes secrets,
  never in ConfigMaps or container images.  Accessed via
  \texttt{LITESTREAM\_ACCESS\_KEY\_ID} and
  \texttt{LITESTREAM\_SECRET\_ACCESS\_KEY} environment variables.

\item \textbf{At-rest encryption}: WAL segments encrypted with
  \texttt{hanzoai/age} (X25519 + ChaCha20-Poly1305).  Even if
  the S3 bucket is compromised, data is unreadable without the
  age secret key.

\item \textbf{Per-org isolation}: Base already encrypts each
  org's SQLite shard with a unique DEK derived via HKDF.
  Replicate streams the encrypted bytes --- it never sees
  plaintext org data.

\item \textbf{Network policy}: The replicate sidecar only needs
  egress to the S3 service (\texttt{s3.zoo.svc.cluster.local:9000}).
  A NetworkPolicy can restrict this.

\item \textbf{Key rotation}: Age key rotation requires a
  re-snapshot.  The apply script can trigger this via
  \texttt{replicate snapshots create}.
\end{enumerate}

\section{Post-Quantum Considerations}

The age encryption layer currently uses X25519 (Curve25519) key
agreement, which is not post-quantum safe.  A sufficiently powerful
quantum computer running Shor's algorithm could recover X25519
private keys from public keys, retroactively decrypting all
age-encrypted S3 objects.

\subsection{Harvest-Now-Decrypt-Later Threat}

S3 replicas are long-lived.  An adversary who captures encrypted
objects today could decrypt them once a cryptographically relevant
quantum computer (CRQC) becomes available.  This
``harvest-now-decrypt-later'' attack is the primary quantum threat
to this replication standard.

\subsection{Mitigation: X-Wing Hybrid KEM}

The mitigation is X-Wing~\cite{xwing}: X25519 + ML-KEM-768 (NIST
FIPS 203~\cite{fips203}).  If quantum computers never materialize,
X25519 provides security.  If they do, ML-KEM-768 provides
post-quantum security.  Both must be broken to compromise the
hybrid.

\texttt{lux/crypto} already implements ML-KEM at
\texttt{encryption/mlkem.go}.  The \texttt{hanzoai/age} fork will
add X-Wing as a new recipient type.

\subsection{Migration Phases}

\begin{description}
\item[Phase 1 (current):] X25519 recipients.  Adequate for
  operational data with limited shelf life.  AES-256 at the
  sqlcipher layer already provides 128-bit post-quantum security
  via Grover's bound.
\item[Phase 2 (planned):] X-Wing hybrid recipients
  (\texttt{age1xwing1...}) in \texttt{hanzoai/age}.  Backward
  compatible---old X25519 recipients continue to work.
\item[Phase 3 (future):] Pure ML-KEM recipients when X25519 is
  deprecated.
\end{description}

\subsection{Three-Layer PQ Status}

\begin{table}[h]
\centering
\caption{Post-quantum status of each encryption layer}
\begin{tabular}{@{}llll@{}}
\toprule
Layer & Algorithm & PQ Status & PQ Path \\
\midrule
Disk & AES-256 (sqlcipher) & Safe (128-bit PQ) & No change \\
S3   & age X25519          & Vulnerable         & X-Wing \\

\end{tabular}
\end{table}
